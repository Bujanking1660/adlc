================================================================================
                    PLUTO PROJECT PLANNER (adlc-planner)
                       DOKUMENTASI DETAIL APLIKASI
================================================================================

1. DESKRIPSI UMUM
   Pluto Project Planner adalah sebuah aplikasi berbasis web (Next.js) yang 
   berfungsi sebagai perancang proyek perangkat lunak berbasis AI. Aplikasi ini 
   memanfaatkan kolaborasi multi-agent (beberapa agen AI yang memiliki peran 
   khusus) untuk menganalisis kebutuhan pengguna (prompt) dan menyusun rencana 
   pengembangan (development plan) serta skema database secara otomatis dan detail.

================================================================================

2. SPESIFIKASI TEKNOLOGI (TECH STACK)
   Aplikasi ini dibangun menggunakan teknologi modern berikut:
   
   - Framework Utama : Next.js 16.2.6 (App Router) & React 19.2.4
   - Bahasa Pemrograman: TypeScript & JavaScript
   - Gaya & Animasi  : TailwindCSS v4 & Lucide React (untuk ikon modern)
   - Database & Auth : Supabase (menggunakan @supabase/ssr & @supabase/supabase-js)
   - Kecerdasan Buatan: Google Generative AI (@google/generative-ai)
     * Model utama yang digunakan: "gemini-3.5-flash"
   - Manajemen State : React State lokal dengan integrasi Supabase secara real-time.

================================================================================

3. STRUKTUR DATABASE (Supabase Schema)
   Supabase digunakan untuk autentikasi pengguna dan penyimpanan data proyek. 
   Terdapat dua tabel utama yang dikonfigurasi dengan Row Level Security (RLS):

   a. Tabel `public.projects`
      Menyimpan informasi proyek yang dibuat oleh pengguna.
      Kolom:
      - `id` (UUID, Primary Key): ID unik proyek.
      - `user_id` (UUID, Foreign Key ke auth.users): Pemilik proyek.
      - `title` (TEXT): Judul proyek.
      - `original_prompt` (TEXT): Deskripsi kebutuhan awal dari pengguna.
      - `final_plan` (JSONB): Menyimpan output final dari QA Agent 
         (plan, schema, schemaFileName, hasBackend, dbType).
      - `created_at` & `updated_at`: Timestamp pembuatan dan pembaruan data.

   b. Tabel `public.agent_logs`
      Menyimpan catatan aktivitas (log) dari setiap agen AI selama proses orkestrasi.
      Kolom:
      - `id` (UUID, Primary Key): ID unik log.
      - `project_id` (UUID, Foreign Key ke public.projects): Proyek terkait.
      - `agent_name` (TEXT): Nama agen (misal: "System Analyst Agent").
      - `status` (TEXT): Status proses ('processing', 'completed', 'waiting_user_input', 'error').
      - `output_data` (JSONB): Menyimpan output sementara atau pertanyaan klarifikasi.
      - `created_at`: Timestamp log.

   * Fitur Keamanan:
     - Row Level Security (RLS) diaktifkan pada kedua tabel.
     - Pengguna hanya diizinkan untuk melihat, menambah, mengubah, dan menghapus 
       data proyek dan log agen milik mereka sendiri (berdasarkan auth.uid()).

4. ALUR KERJA ORKESTRASI MULTI-AGENT (/src/app/api/orchestrate/route.ts)
   Orkestrasi proses perencanaan dibagi menjadi 4 tahap utama yang dijalankan secara 
   berurutan (pipelined) dengan dukungan interaksi pengguna:

   TAHAP 1: System Analyst Agent ('analyst')
   - Menganalisis prompt awal pengguna.
   - Memastikan apakah prompt berkaitan dengan pengembangan software. Jika tidak, 
     menghasilkan status 'out_of_scope' dan menghentikan proses.
   - Jika berkaitan tetapi terlalu ambigu, menghasilkan status 'waiting_user_input' 
     dengan daftar 1-3 pertanyaan klarifikasi untuk dijawab oleh pengguna.
   - Jika sudah cukup jelas, menghasilkan Rencana Analisis Sistem awal.

   TAHAP 2: UI/UX Frontend Agent ('frontend')
   - Mengambil konteks dari prompt asli dan hasil analisis sistem.
   - Menganalisis kebutuhan antarmuka (layout, warna, komponen).
   - Menghasilkan pertanyaan klarifikasi kontekstual yang spesifik (bukan template 
     umum) jika ada detail desain penting yang kurang jelas.
   - Jika sudah jelas, menghasilkan Rencana Frontend (desain halaman dan UI).

   TAHAP 3: Database Backend Agent ('backend')
   - Menentukan apakah aplikasi membutuhkan database atau backend.
   - Jika aplikasi hanya membutuhkan state lokal (misalnya kalkulator sederhana), 
     backend dinyatakan tidak diperlukan.
   - Jika membutuhkan backend, agen akan menanyakan detail spesifik arsitektur data 
     atau integrasi jika dirasa kurang jelas.
   - Jika sudah jelas, menghasilkan Rencana Backend & Struktur Data.

   TAHAP 4: QA & DevOps Agent ('qa') - Tahap Akhir
   - Menggabungkan rencana dari analis, frontend, dan backend.
   - Menentukan jenis database (SQL seperti PostgreSQL/SQLite, atau NoSQL seperti MongoDB).
   - Menghasilkan file rencana komprehensif (`plan.md`) dan skema database terpisah 
     (misalnya `schema.sql` atau model NoSQL).
   - Menyimpan hasil akhir ke Supabase pada kolom `final_plan` di tabel `projects`.

5. MEKANISME ROTASI API KEY GEMINI
   Untuk mengatasi keterbatasan kuota rate limit dan menjaga stabilitas aplikasi, 
   sisi backend mengimplementasikan sistem rotasi API Key secara acak (stateless 
   random rotation) di file `/src/app/api/orchestrate/route.ts`.
   API Key yang dapat digunakan dideklarasikan di file `.env.local`:
   - `GEMINI_API_KEY_1`
   - `GEMINI_API_KEY_2`
   - `GEMINI_API_KEY`
   Sistem akan memfilter key yang valid, memilih secara acak untuk setiap request, 
   dan mencatat key mana yang digunakan ke konsol server.

   │  │  ├─ layout.tsx        -> Root layout aplikasi dengan tema global.

   │  │  └─ supabase.ts       -> Inisialisasi Supabase client (Browser & Admin Server).
   │  └─ components/

   │     ├─ AuthUI.tsx        -> Form login/register Supabase Auth & tombol Guest Mode.
   │     ├─ Workspace.tsx     -> Workspace utama manajemen proyek dan input prompt.
   │     ├─ PromptInput.tsx   -> Input area prompt awal dengan rekomendasi cepat.
   │     ├─ AgentProgress.tsx -> Visualisasi langkah/status agen AI yang berjalan.
   │     ├─ InteractiveModal.tsx -> Modal interaktif untuk menjawab pertanyaan klarifikasi agen.
   │     ├─ PlanViewer.tsx    -> Panel untuk membaca Markdown rencana pengembangan (plan.md).
   │     └─ PlanCanvas.tsx    -> Kanvas visualisasi skema database/tabel secara interaktif.

7. FITUR UNGGULAN APLIKASI
   1. Dynamic Clarification Loop: AI tidak langsung menebak, melainkan bertanya 
      secara cerdas apabila kebutuhan proyek masih kurang jelas di setiap tahapan.
   2. Guest Mode (Local Mode): Pengguna dapat mencoba aplikasi dan mendesain 
      proyek tanpa perlu mendaftar akun terlebih dahulu.
   3. Dual-Theme Support: Dilengkapi dengan Dark Mode dan Light Mode adaptif yang 
      sangat ramah di mata.
   4. Database Visualizer (Plan Canvas): Secara visual menampilkan tabel dan 
      relasi kolom dari skema database SQL/NoSQL yang dihasilkan oleh AI.
   5. Real-time Agent Log Tracker: Pengguna dapat memantau apa yang sedang dipikirkan 
      atau dikerjakan oleh masing-masing agen AI secara live.
